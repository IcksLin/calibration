# Perspective Matrix Calibrator

用于标定全局逆透视矩阵的独立工具工程。工具以图形界面方式工作，支持四点选取、二值辅助、虚拟矩形构造、矩阵实时输出、预览图保存、会话保存/恢复和 Windows 单文件打包。

## 项目目标

本工具的目标不是把“四边形内部裁出来拉正”，而是基于用户选取的参考四边形，构造一个虚拟目标矩形，进一步求出整幅图像的全局逆透视矩阵，并实时显示逆透视后的完整视野效果。

输出结果可直接复制到现有嵌入式工程中使用，格式与工程中的 `cv::Mat M`、`cv::Mat M_Reverse` 以及 `#define M2PIX` 保持一致。

## 工程结构

```text
perspective_calibrator/
├─ app.py
├─ core.py
├─ services.py
├─ build_windows.py
├─ assets/
│  └─ anan.jpg
├─ dist/                 # 打包输出目录，非源码必需
├─ build/                # PyInstaller 临时目录，非源码必需
├─ build_spec/           # PyInstaller spec 输出目录，非源码必需
└─ runtime/              # 源码运行时生成的日志/图片目录
```

各模块职责如下：

- `app.py`
  负责界面显示、鼠标交互、参数控件、缓冲层调度、日志管理、矩阵文本输出和预览图保存触发。

- `core.py`
  负责纯几何与数学逻辑，包括四点排序、虚拟矩形构造、单应矩阵求解、工程格式输出和 `M2PIX` 计算。

- `services.py`
  负责功能层图像处理，包括图像载入、灰度/二值缓存生成、全局逆透视预览计算以及预览图落盘。

- `build_windows.py`
  负责将工具打包为 Windows 单文件 `exe`，并自动把 `assets/anan.jpg` 转换为可执行文件图标。

- `assets/`
  工具本地资源目录，目前包含窗口/打包 Logo。

- `runtime/`
  源码运行时生成的目录。默认会在这里创建：
  - `runtime/Picture/`：保存的逆透视预览图
  - `runtime/logs/calibrator.log`：滚动日志文件

## 总体框架

当前工程采用 `外界交互 -> 缓冲层 -> 内核/功能层` 的结构，目的是隔离用户高频拖动输入与图像计算的有限处理能力，避免高频重算导致界面卡死。

### 1. 显示层

显示层位于 `app.py`，主要包含：

- 左侧控制面板
- 左侧原图/二值辅助图显示区
- 右侧全局逆透视预览区
- 点位、虚拟矩形、矩阵文本和保存按钮等 UI 元素

显示层只负责采集用户动作和展示结果，不直接做重图像计算。

### 2. 缓冲层

缓冲层也在 `app.py` 中，由以下状态和函数组成：

- `_pending_kernel_updates`
- `_buffer_frame_interval_ms`
- `_schedule_kernel_flush()`
- `_flush_kernel_buffer()`
- `_set_kernel_busy()`

缓冲层特性：

- 用户拖动滑条时，只是把最新值覆盖写入缓冲区
- 缓冲层按固定帧率择机向内核提交
- 如果内核正忙，则当前帧跳过，等待下一帧再提交
- 多次高频更新会自动合并，只保留最新值

### 3. 内核层

内核层分成两部分：

- `core.py`：几何求解内核
- `services.py`：图像处理功能内核

这两层负责：

- 构造虚拟矩形
- 计算 `M` / `M_Reverse`
- 计算 `M2PIX`
- 生成左图显示缓存
- 生成右侧全局逆透视预览图

### 4. 功能层

功能层基于当前稳定内核状态执行额外动作，例如：

- 保存预览图
- 导出矩阵文本
- 保存/读取会话
- 记录运行日志

## 运作流程

下面是工具的实际运行流程。

### 1. 图像导入

用户点击“选择图像”后：

1. `app.py` 调用 `CalibrationService.load_image()`
2. `services.py` 生成 RGB、灰度图、二值图缓存
3. 显示层刷新左图
4. 控件范围根据图像尺寸自动更新

### 2. 用户选四点

用户在左图依次点击四个点后：

1. 点位写入 `CalibrationState.points`
2. `core.py` 对四点自动排序为左上、右上、右下、左下
3. 左侧数据区同步显示排序后的点位角色和坐标
4. 左图叠加绘制四边形和虚拟矩形辅助框

### 3. 参数调整

用户调节长宽比、底边距离、水平偏移、底面边长度缩放、阈值等参数时：

1. 交互层把最新值写入缓冲区
2. 缓冲层以固定帧率尝试提交
3. 若内核空闲，则批量应用最新参数
4. 若内核繁忙，则等待下一帧继续尝试

这样可以保证：

- 拖动滑条时界面依然流畅
- 高频输入不会强制每一步都做一次重计算
- 程序更不容易因重算过密而卡死

### 4. 构造虚拟矩形

内核在 `core.py` 中根据四点和参数生成虚拟目标矩形：

- 底边参考近端地面边
- 宽度受“底面边长度缩放”影响
- 高度由矩形长宽比推导
- 底边到图像底部的距离由“虚拟底边距图像底部”控制
- 左右位置由“虚拟矩形水平偏移”控制

### 5. 计算矩阵并生成预览

随后内核执行：

1. 根据原始四边形和虚拟矩形求解正向矩阵 `M`
2. 对 `M` 求逆得到 `M_Reverse`
3. 使用逆矩阵对整幅图像做全局逆透视采样
4. 刷新右侧预览图
5. 刷新矩阵文本框与 `M2PIX`

### 6. 保存预览图

点击“保存预览图”时：

1. 保存请求先进入缓冲层
2. 缓冲层在合适时机把命令提交给功能层
3. 功能层取当前稳定的预览图快照
4. 保存到 `Picture/时间戳.png`
5. 文件句柄立即释放，避免文件被长期占用

## 运行方式

### 源码运行

在工具目录或工程根目录下都可以运行。

示例一：在工程根目录执行

```bash
python calibration/perspective_calibrator/app.py --image shot.png
```

示例二：先进入工具目录再执行

```bash
python app.py --image shot.png
```

不带 `--image` 也可以，程序启动后可从界面中手动选择图像。

### 打包为 Windows 可执行文件

```bash
python calibration/perspective_calibrator/build_windows.py
```

打包完成后，生成文件位于：

```text
calibration/perspective_calibrator/dist/PerspectiveMatrixCalibrator.exe
```

如果旧的 `PerspectiveMatrixCalibrator.exe` 正被系统占用，打包脚本会自动回退到带时间戳的新目录，例如：

```text
calibration/perspective_calibrator/dist_20260419_231601/PerspectiveMatrixCalibrator.exe
```

### EXE 运行时输出目录

打包后的 `exe` 运行时，运行产物默认写到 `exe` 同级目录：

- `Picture/`：保存的逆透视预览图
- `logs/calibrator.log`：滚动日志

这样可以避免 PyInstaller 临时解包目录导致的文件丢失或路径混乱问题。

## 独立剥离说明

当前工程已经支持“整文件夹剥离”。

也就是说，直接复制整个 `perspective_calibrator` 文件夹到其他目录后，源码模式仍可正常运行。原因如下：

- `app.py` 和 `services.py` 已支持独立目录导入
- Logo 已内聚到 `assets/anan.jpg`
- 打包脚本优先使用本地 `assets/`
- 运行日志和图片保存路径都按工具目录或 `exe` 同级目录解析

注意：

- 需要复制整个 `perspective_calibrator` 文件夹
- 不建议只单独复制 `app.py`

## 会话与输出

### 会话文件

界面支持保存/加载 `json` 会话文件，会记录：

- 图像路径
- 阈值
- 长宽比
- 虚拟矩形位置参数
- 底面边长度缩放
- 实际物理宽度
- 已选四点

### 工程输出

矩阵文本框会实时输出：

- `cv::Mat M`
- `cv::Mat M_Reverse`
- `#define M2PIX`

输出格式与现有工程代码兼容，可直接复制使用。

## 日志机制

程序会将关键事件写入滚动日志文件，例如：

- 启动
- 打开图像
- 选点
- 撤销/清空
- 保存会话
- 载入会话
- 保存预览图
- 未捕获异常

日志文件会自动裁剪，只保留最近一段内容，避免长时间运行后无限增长。

## 维护建议

- 新增参数时，优先接入缓冲层，不要让控件直接高频调用重计算逻辑
- 新增输出功能时，尽量通过功能层执行，不要直接在 UI 回调里做重 I/O
- 如果未来功能继续增长，可以把当前的轻量缓冲调度进一步抽象成独立 `CommandHub`
